# 0.1.9 스플래시에서 멈추는 문제 (마이그레이션 실패 + 오류 은폐)

## 날짜
2026-07-21

## 증상

0.1.9를 설치해 실행하면 스플래시가 "Initializing configuration..."에서 멈추고
진행되지 않는다. 사용자 보고 오류 문구: **"cannot roll back, no transaction is
active"**.

## 원인 두 가지

### 1. 오류가 스플래시 뒤에 가려진다 (멈춘 것처럼 보이는 이유)

`setup.initialize()`가 예외를 던지면 `main.py`가 이를 잡아
`QMessageBox.critical(...)`로 알리게 되어 있다. 그런데 스플래시는
`Qt.WindowStaysOnTopHint`라 **오류 대화상자가 스플래시 뒤에 열린다.** 사용자
눈에는 실패한 단계에서 얼어붙은 스플래시만 보이고, 그 뒤에서 모달 대화상자가
응답을 기다리므로 프로세스도 그대로 서 있는다.

즉 "멈춤"이 아니라 **보이지 않는 오류 대화상자**였다.

### 2. 진짜 원인: 마이그레이션 파일의 비ASCII 문자

로그(사용자 제공)가 결정적이었다:

```
File "peewee_migrate\router.py", line 293, in read
UnicodeDecodeError: 'cp949' codec can't decode byte 0xe2 in position 94
```

`peewee_migrate`의 `Router.read`는 마이그레이션 파일을 **인코딩 지정 없이**
연다:

```python
with path.open("r") as f:   # 플랫폼 기본 인코딩
```

한국어 Windows의 기본 인코딩은 cp949다. 내가 006의 docstring에 쓴 em dash
(`—`, UTF-8로 `0xe2 0x80 0x94`)가 여기서 디코드되지 않는다. 확인해 보니
001~005는 비ASCII 바이트가 **0개**인데 006만 6개였고, 첫 위치가 로그의
`position 94`와 정확히 일치했다.

즉 **한국어 Windows 사용자는 신규 설치든 업그레이드든 앱이 시작되지 않는다.**
개발 환경(WSL, UTF-8 기본)에서는 재현되지 않아 릴리스까지 통과했다.

이 위에 앞서 말한 rollback 은폐가 겹쳐, 진짜 원인이 로그에서 사라졌다.

### 곁가지: 중단된 실행

원인 조사 중 별개 취약점도 확인했다. `peewee_migrate`는 마이그레이션 성공
후에야 `migratehistory`에 기록하므로, 중단된 실행은 **컬럼은 추가됐지만 기록은
없는** 상태를 남길 수 있고 재시도가 `duplicate column name`으로 실패한다.
이번 장애의 원인은 아니지만 같은 은폐 문제를 유발하므로 함께 방어했다.

## 수정

### `migrations/006_20260721.py` — ASCII로, 그리고 멱등하게

1. **비ASCII 제거** (실제 수정). em dash를 콜론으로 바꾸고, 파일 상단에 왜
   ASCII를 유지해야 하는지 남겼다. cp949 로케일을 흉내 낸 재현 테스트로
   확인했다 (인코딩 미지정 `open`이 cp949를 쓰도록 강제 → 마이그레이션 성공).
2. **멱등 처리** (곁가지 방어). 적용 전에 `PRAGMA table_info`로 컬럼 존재를
   확인하고 이미 있으면 건너뛴다. rollback도 대칭. fake 모드에서는 데이터베이스가
   mock이라 조회를 건너뛴다.

### `main.py` — 실패를 보이게

- 오류 대화상자를 띄우기 전에 스플래시를 닫는다.
- `splash` 변수를 `try` 블록 **앞에서** 바인딩한다. 이전에는 블록 안에서
  정의돼, 그 지점 이전에 실패하면 예외 처리기가 `NameError`를 낼 수 있었다.
- 대화상자에 로그 디렉터리 경로를 함께 표시해 다음 진단을 쉽게 했다.

검증: `initialize()`가 실패하도록 강제하고 실행 → 대화상자가 표시되고, 보이는
스플래시가 0개로 남지 않음을 확인.

### `tests/test_migrations.py` (신규)

재발 방지가 핵심이다. 이 버그는 개발 환경에서 재현되지 않으므로 파일 자체를
검사해야 한다:

- **마이그레이션 파일이 순수 ASCII인지** (파일별 파라미터화). 이번 버그를
  바로 잡아낸다.
- 빈 데이터베이스에 전체 체인이 적용되는지 (신규 설치 경로).
- 중단된 실행 이후 재시도가 통과하는지 (멱등성).
- `chart_settings_json` 컬럼이 실제로 생기는지.

## 교훈

- **마이그레이션 파일은 ASCII만.** 라이브러리가 로케일 기본 인코딩으로 읽는다.
- **개발 환경이 UTF-8이라 이 계열 버그는 통과한다.** 로케일 의존 코드는 테스트로
  고정해야 한다.
- `peewee_migrate`가 rollback 실패로 원래 예외를 덮어쓰는 동작은 우회할 수 없다.
  그래서 마이그레이션 실패의 진단 비용이 크다 — 애초에 실패하지 않게 방어할 것.

## 사용자 조치

다음 빌드를 설치하면 해결된다. 데이터베이스를 손볼 필요는 없다 (006이 적용된
적 없으므로 기존 데이터는 그대로다).
